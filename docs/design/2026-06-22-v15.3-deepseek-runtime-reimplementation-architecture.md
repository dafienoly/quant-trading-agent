# V15.3 DeepSeek Agent Runtime 重实现架构

## 1. 模块边界

```text
src/llm/
  schemas.py          请求、结果、profile 与业务输出 schema
  deepseek_runtime.py async-first 调用、超时、重试、工具循环与校验
  conversation.py     多轮消息、脱敏持久化与 TTL
  context_cache.py    稳定 system/schema 前缀与指纹
  tool_registry.py    只读工具白名单、参数和路径安全
  usage.py            非敏感调用指标
  model_router.py     配置入口和旧业务兼容委托
```

BugFixAgent 只依赖 `DeepSeekRuntime` 和 schema，不再依赖 OpenAI SDK。

## 2. 调用流程

```text
业务 Agent
  -> ModelRouter 或 DeepSeekRuntime
  -> 校验 profile / schema / tools
  -> ContextPrefixCache 生成稳定前缀
  -> AgentConversation 组装消息
  -> AsyncOpenAI + timeout + semaphore
  -> 只读工具循环（任一错误立即终止）
  -> JSON 解析
  -> schema 校验并规范化
  -> UsageTracker
  -> DeepSeekResult
```

只有 `status=ok` 且 `data` 已通过 schema 校验时，业务层才能继续处理。

## 3. 错误与重试

| 场景 | 状态 | 是否重试 |
|---|---|---|
| 缺密钥、缺 SDK | `unavailable` | 否 |
| API timeout | `timeout` | 是，最多配置次数 |
| 连接、限流、服务端瞬时错误 | `api_error` | 是 |
| 参数、鉴权及其他非瞬时错误 | `api_error` | 否 |
| 空内容、非法 JSON、未知或不匹配 schema | `invalid_response` | 否 |
| 工具越权、未知、参数错误、执行错误、轮次超限 | `tool_error` | 否 |
| asyncio cancellation | 向调用者传播 | 否 |

错误结果只返回脱敏原因和短摘要，不包含 key、完整 traceback 或 reasoning content。

## 4. Schema 策略

- Runtime 维护明确的 schema registry。
- BugFix analysis/proposal 使用严格 Pydantic 模型，禁止额外字段；风险等级使用枚举。
- `needs_human_review` 和 `requires_approval` 固定为 `true`。
- 因子发现、研究推荐和信号解释使用各自的顶层结构模型。
- 未注册 schema 默认拒绝，不允许“无 schema 也成功”。
- schema 的 JSON 描述进入稳定 prompt prefix，并用于提示模型返回 JSON。

## 5. 工具安全

- 注册器只接受 `read_only=true` 工具。
- Runtime 在发请求前校验请求工具全部存在且 profile 允许。
- 参数按工具 JSON schema 校验 required、类型和额外字段。
- 文件读取必须位于项目允许目录并经过 `resolve()`，拒绝路径逃逸和敏感目录。
- 搜索工具使用固定 `rg` 可执行文件、固定参数和超时，不开放任意命令。
- 工具结果执行截断和密钥脱敏；任一工具错误立即返回 `tool_error`。

## 6. 会话与缓存

- 会话 ID 使用 UUID，不依赖进程内对象地址。
- 会话默认保存在 `runtime/llm_conversations/`，TTL 为 7 天。
- 保存前递归脱敏，默认移除 `reasoning_content`。
- ContextPrefixCache 只保存系统不变量、profile 指令和 schema，不保存用户 prompt。
- Runtime 实际使用缓存 entry 的 prefix 作为 system message。

## 7. BugFixAgent 迁移

- public API `analyze()`、`propose_fix()`、`execute_fix()` 保持不变。
- analyze/propose 使用不同 profile 和严格 schema。
- Runtime 失败直接返回结构化失败，不再把空或非法内容包装成成功字典。
- proposal 通过 schema 后仍执行现有受限模块检查。
- execute_fix 的人工审批、工作区保护、测试和回滚逻辑不变。

## 8. 测试策略

- fake client 覆盖请求参数、JSON、Thinking、tool loop 和 usage。
- 独立测试 timeout、retry、cancel、concurrency。
- 会话测试覆盖脱敏、reasoning 不持久化和 TTL。
- 工具测试覆盖路径逃逸、未知工具、写工具、参数错误、敏感输出。
- BugFixAgent 测试覆盖 Runtime 委托、失败关闭、受限模块与审批字段。
- 搜索确认 `chat.completions.create` 只存在于 Runtime。
