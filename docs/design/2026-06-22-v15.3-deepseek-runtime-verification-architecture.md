# V15.3 DeepSeek Agent Runtime 功能真实性验证设计

## 1. 设计目标

本任务采用只读代码审计、提交关系核对和离线测试相结合的方式，回答一个限定问题：PR #3 声称的 DeepSeek Agent Runtime 与 BugFixAgent 迁移是否已经真实存在于当前 `main`。

验证本身不改变产品代码，不把旧 PR 内容引入当前分支，也不访问真实模型服务。

## 2. 基线与对照

### 2.1 产品基线

- 当前 `main` 是唯一验收基线。
- 当前功能分支从最新 `main` 创建，只新增 V15.3 验证文档。

### 2.2 对照来源

- PR #3 的元数据、描述和 head tree 用于确认其声明与旧实现位置。
- 原始需求 `docs/requirements/2026-06-12-deepseek-agent-runtime-requirements.md` 用于恢复目标能力。
- 原始架构 `docs/design/2026-06-12-deepseek-agent-runtime-architecture.md` 用于核对预期模块和调用边界。

对照来源不能替代主线实现证据。

## 3. 证据链

### 3.1 提交关系证据

核对 PR #3 的状态、base/head、与当前 `main` 的 ahead/behind 数量、merge-base，以及 PR head 是否被 `main` 包含。若 PR head 未被 `main` 包含，则继续检查能力是否由其他提交等价实现。

### 3.2 静态实现证据

检查以下位置及其等价实现：

- `src/llm/`
- `src/product_app/bug_fix_agent.py`
- Runtime 相关环境变量和配置
- OpenAI 兼容 client 的直接调用点
- LLM 到交易模块的导入和调用边界

### 3.3 测试证据

运行现有 ModelRouter 与 BugFixAgent 单元测试，确认当前代码仍可执行；同时审计测试名称和断言，判断 13 项目标是否有针对性覆盖。现有测试通过只证明当前行为稳定，不自动证明缺失能力已经实现。

### 3.4 失败路径证据

- 缺少密钥：检查现有测试及返回状态。
- 请求超时：检查 client 调用是否有显式 timeout，以及异常是否转换为可诊断的拒绝结果。
- 空内容：调用现有解析入口，确认空内容是否被拒绝，还是被包装成可继续流转的普通结果。

## 4. 结论分级

| 结论 | 条件 |
|---|---|
| `PASS` | 13 项全部满足，且核心项具有自动化测试 |
| `PASS_WITH_NOTES` | 13 项满足，仅存在非阻断证据或文档缺口 |
| `REJECTED` | 任一核心能力未进入 `main`，或 fail-closed / 安全边界不成立 |

本任务不以“旧 PR 曾实现”降低验收标准。

## 5. 未实现时的后续方案

若结论为 `REJECTED`：

1. 保留本次验证报告作为新的需求输入。
2. 从最新 `main` 新建实现分支，不直接合并 PR #3。
3. 按原始需求重新设计 Runtime 公共接口，并补齐 fail-closed 契约。
4. 可人工参考 PR #3 的概念和测试思路，但逐项重新审查与当前主线的兼容性。
5. 在新实现 PR 中提供中文功能说明报告、中文验收报告和完整离线测试证据。

## 6. 安全边界

- 不接触交易敏感模块。
- 不引入真实下单、自动交易或 LLM 买卖决策能力。
- 不运行真实模型请求。
- 不提交 `.agent/tmp/**`、`.agent/reports/**` 或本地测试临时目录。
- 不自动合并 `main`。
