# V15.2 Pipeline Dashboard Artifacts 需求文档

## 背景

当前 Agent Pipeline 已有阶段门禁、回归脚本和静态 HTML 报告查看器，但 GitHub Actions 尚未把这些能力连成稳定的 PR 验证链路。Draft PR 或普通功能 PR 可能出现 `gh pr checks` 返回无检查记录，失败运行也缺少统一的 JSON 与 HTML 诊断产物。

## 目标

1. Pipeline 阶段和 PR 验证结束后生成 `.agent/reports/pipeline_report.json` 与 `.agent/reports/pipeline_dashboard.html`。
2. GitHub Actions 在成功和失败时都上传 dashboard artifact。
3. PR conversation 提供 artifact 查看说明，并避免同一 PR 被重复评论刷屏。
4. 新增 `pull_request` 触发的轻量 PR validation workflow。
5. 保持 main 只能人工审阅、人工合并。

## 非目标

1. 不实现远程托管 dashboard 服务。
2. 不修改交易、风控、订单、账户、券商或真实交易能力。
3. 不把 `.agent/reports/**` 提交到仓库。
4. 不改变现有 Agent 阶段职责和报告门禁语义。

## 功能需求

| ID | 功能 | 预期行为 | 验收标准 |
|---|---|---|---|
| F-001 | Pipeline 报告产物 | 回归结果写入 JSON，并渲染静态 HTML dashboard | 成功运行同时生成两个非空文件 |
| F-002 | 失败诊断产物 | 前置步骤失败时仍执行报告生成与 artifact 上传 | workflow 使用 `always()`，缺失正常报告时生成诊断性 fallback 文件 |
| F-003 | Artifact 上传 | stage runner、merge gate 和 PR validation 上传报告 | workflow 使用 `actions/upload-artifact@v4`，且 `if-no-files-found` 不静默成功 |
| F-004 | PR 查看说明 | PR conversation 提供 artifact 名称和 Actions 查看路径 | 评论为中文，带稳定 marker，可更新已有评论 |
| F-005 | PR validation | `pull_request` 事件触发轻量检查 | Draft、Ready、同步提交等 PR 生命周期事件至少产生一个 check run |
| F-006 | 回归检查 | 运行严格 pipeline 回归与两份目标 pytest | 命令与用户指定命令一致，失败返回非零 |
| F-007 | Git 与敏感路径检查 | 检查 diff 格式、受限模块和临时产物跟踪状态 | 任一检查失败时 workflow fail closed |
| F-008 | 中文界面 | Dashboard 与新增 PR 评论使用中文 | HTML 标题、栏目和状态说明为中文 |
| F-009 | 手工合并 | 自动化不调用 PR 自动合并 | workflow 中不存在 `gh pr merge` |

## PR Validation 最低命令

```bash
python scripts/agent_pipeline_regression.py --strict
python -m pytest tests/test_agent_pipeline_automation.py tests/test_agent_pipeline_regression.py -q
git diff --check
git diff --name-only origin/main...HEAD
git ls-files .agent/tmp .agent/reports
```

PR validation 的回归命令允许增加 `--output` 等报告参数，但不得降低 `--strict` 语义。pytest 可增加隔离用 `--basetemp` 参数。

## 安全约束

1. 受限路径至少覆盖 `src/broker/**`、`src/execution/**`、`src/order/**`、`src/account/**`、`src/risk/**`、对应 `*_engine` 路径、`miniQMT/**`、真实交易和真实订单提交相关路径。
2. 不允许 fallback、mock、smoke 或 handoff-preview 产物冒充正式验收报告。
3. `real=true` 且 `strict=true` 的现有 fail-closed 行为不得削弱。
4. `.agent/reports/**` 与 `.agent/tmp/**` 必须被忽略且不得有已跟踪文件。
5. 不自动合并 main，PR 必须人工审阅。

## 可观测性

JSON 报告至少包含总体状态、检查摘要、逐项检查和 artifact 元数据。HTML dashboard 至少展示总体状态、失败项、阶段/门禁信息、文档清单、受限路径检查和临时目录卫生状态。

## 测试范围

1. Dashboard viewer 的中文渲染、JSON 输入校验和 HTML 输出。
2. PR validation workflow 的触发条件、命令、受限路径检查和 tracked-file 检查。
3. Stage runner 与 merge gate 的 `always()` 报告生成、artifact 上传和 PR 评论。
4. 现有 Agent Pipeline regression strict 模式。
5. 交易敏感模块 diff 必须为空。
